What is Harness Engineering?

Harness Engineering is an orchestration framework for building software with AI agents. Instead of developers writing code directly, agents handle implementation while humans orchestrate, review, and guide quality through structured checkpoints called gates.

The harness tracks every task from spec to completion, enforces quality gates, and captures lessons learned so agents improve with each iteration.

🎯
Spec-Driven
Every task starts with a detailed spec. Agents build from specs, not vague descriptions. The spec is the contract.
🚧
Gate-Enforced
Quality checkpoints between each phase. Unit tests, visual verification, human review, E2E testing. No shortcuts.
🔄
Learning System
Every bug found becomes a test case. Every meeting becomes lessons. Skills and rules evolve with each story.

Repositories

Three repos, each with a distinct role. The harness is the command center. Code repos are where agents work.

planyear-harness
Command center. CLI tool, harness.json (source of truth), task specs, gate definitions, lessons learned, memory files. Humans orchestrate from here.
platform
Backend. Java/Spring Boot microservices, MongoDB. Agents create worktrees here for BE development. Has its own .claude/skills for BE-specific rules.
unified-portal
Frontend. React/TypeScript SPA. Agents create worktrees here for FE development. Has its own .claude/skills for FE-specific rules.

Roles

👤
Human Orchestrator
Writes/reviews specs, triggers commands, reviews gates, makes scope decisions. Never writes production code directly. Works from the harness repo.
🤖
Development Agent
Builds features from task specs in isolated worktrees. Writes code, tests, and Storybook stories. Works in platform or unified-portal repos.
⚙️
Harness CLI
Orchestration tool. Creates worktrees, tracks status, enforces gates, validates specs, manages phase transitions. The guardrails.

Task Lifecycle

Every task follows this flow. Coloured by who owns each step.

prep Generate spec template + prompt. Creates scaffolding and fixture directories.
→
Write Spec Human fills in the spec (or uses the generated prompt). Must pass spec-check before starting.
→
spec-check Validates spec completeness: summary, scope, gates, visual refs, no unanswered [HUMAN INPUT REQUIRED] markers.
→
start Creates git worktree, generates agent prompt, sets status to in_progress. Runs spec-check automatically.
→
Agent Builds Agent implements the feature in the worktree. Writes code, unit tests, integration tests, Storybook stories.
→
Gates 1-3 Unit tests, visual verification (Figma + Storybook), human review. Each gate must pass before proceeding.
→
pr-ready Marks task as in_review, records PR URL. Code is ready for peer review.
→
mark-merged Marks task as merged, removes worktree, syncs progress.
→
Human Review Human verifies the feature in a running system. Logs bugs with ./harness bug. Every bug becomes a test case.
→
Gate 4: E2E End-to-end testing via Playwright. Story-scoped scenarios. Flapdoodle isolation. Agent runs headless, human runs headed.
→
Complete All gates passed. Story is done. Lessons captured. Ready for next story.

Phases

Work is organised into phases. Each phase is a self-contained batch of tasks. Tasks within a phase can run in parallel. Tasks across phases cannot.

PhasePurposeParallelism
Phase 0Infrastructure and test setup. Unit test scaffolding, Storybook setup, Playwright smoke test, MSW handlers, integration test base.Sequential (foundational)
Phase 1Core features. The main stories for this EPIC. BE and FE development with full gate enforcement.Parallel within phase
Phase 2Secondary features. Builds on Phase 1 output.Parallel within phase
Phase 3Edge cases, migration, cleanup.Parallel within phase

Phase transitions happen via ./harness phase-transition N. All tasks in the current phase must be complete before transitioning. The baseline fixture is versioned at each phase boundary.

Modes

Each task has a mode that determines how BE and FE work is coordinated.

Mode A Parallel
API contract is locked upfront. BE and FE agents work simultaneously. Both can start from the same spec. Faster but requires a stable contract.
Mode B Sequential
BE goes first, discovers the API contract. FE starts after BE gates pass. Safer for complex or uncertain features. The contract emerges from BE work.
FE-only
No backend changes needed. Frontend-only task. Only FE gates apply.

Streams

Tasks are grouped into streams based on the feature area they affect.

SR Start Renewal
The renewal initiation flow. Tier validation, plan selection, renewal configuration.
TS Tier Structure
Tier mismatch detection, auto-conversion, quote processing, offer display.
BC Benefit Class
Plan explosion by benefit class, grouped plan management, contribution modelling.

Gates

Quality checkpoints that must pass before work proceeds. Each gate has specific checks (automated tests, visual review, human verification).

🧪
test
Automated. Runs specific test commands (unit, integration). Pass/fail based on exit code.
📖
storybook
Visual. Agent compares Storybook output against Figma design. Figma MCP for token verification.
👤
human
Manual. Human reviews and provides structured feedback. All issues listed at once, not one by one.
🌐
e2e
End-to-end. Story-scoped Playwright scenarios. Flapdoodle isolation. Fresh data per run.
How gates are enforced

Gates are defined in harness.json per task, each with a list of typed checks. The ./harness check-gate command runs automated checks, prompts for manual ones, and records the result.

Agents cannot self-report gate completion. They must list specific test commands run and their output. This prevents the self-evaluation bias where agents confidently claim success without evidence.

CLI Commands

All commands run from the harness repo root: ./harness <command>

CommandWhat it doesWhen to use

All commands support --dry-run to preview without making changes. Use --side be|fe for tasks with both backend and frontend.

Common Workflows

Start a new task

1. ./harness prep PY-XXXXX (generate template + prompt)

2. Paste the prompt into your Claude session to fill the spec

3. Review the spec, fill any remaining [HUMAN INPUT REQUIRED]

4. ./harness spec-check PY-XXXXX (validate)

5. ./harness start PY-XXXXX --side be (create worktree, start agent)

Run through gates

1. Agent completes development in worktree

2. ./harness gates PY-XXXXX (see what gates exist)

3. ./harness check-gate PY-XXXXX 1 (run Gate 1 checks)

4. If failed: fix issues, re-run. If passed: move to next gate.

5. Repeat for each gate until all pass.

Complete a task

1. ./harness pr-ready PY-XXXXX --side be --pr-url "platform #151"

2. Peer reviews and approves PR

3. ./harness mark-merged PY-XXXXX --side be --pr-url "platform #151"

4. Human verifies in running system, logs bugs with ./harness bug

5. E2E gate passes

6. Update harness.json status to "complete"

Fix-forward (FE finds BE bug)

If the FE agent discovers a missing API field or BE contract issue:

1. FE agent creates a small targeted BE fix in a new branch

2. Scoped to only the missing field or transform

3. Raises a small BE PR alongside the FE work

4. Keeps the FE agent moving instead of waiting for coordination

Session Model

Keep separate Claude sessions for different concerns. Never mix them.

Command Center Session
Stays in the harness repo. Runs CLI commands, reviews specs, tracks progress. Uses Claude Code desktop (stays anchored). Never cd's to code repos.
BE Agent Session
Works in a platform worktree. Reads task spec from harness (absolute path). Writes code, tests. Never modifies harness files by changing directories.
FE Agent Session
Works in a unified-portal worktree. Same rules as BE session. Uses FE-specific skills and rules.

Critical: CLI agents tend to cd into different directories, losing access to CLAUDE.md and skills. Always instruct agents to use absolute paths without changing directories.

Test Data and Fixtures

Two-layer fixture system for E2E testing. Each run starts fresh (Flapdoodle, no shared DB).

Baseline Fixture
Shared across all stories. Broker org, employers, users, system plans. Loaded by SeedDataService on Flapdoodle startup. Versioned at each phase transition.
Story-Specific Fixture
Per-task additions on top of baseline. Upload files (quotes, documents), extra seed data. Lives in the task spec directory under fixtures/ and seed/.
